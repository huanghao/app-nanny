# app-nanny 设计文档

> 本地开发服务统一管理工具
> CLI 命令名：`nanny`
> 日期：2026-06-04

---

## 1. 背景与目标

### 痛点

本地同时开发多个项目时，每个项目通过 `just dev` 启动若干服务，带来以下问题：

- **端口冲突**：新项目默认占用相同端口，与已有服务冲突
- **端口失忆**：记不清哪个服务跑在哪个端口
- **终端堆积**：每个服务占用一个终端窗口，切换混乱
- **日志黑洞**：服务放到后台跑，错误日志看不到
- **Agent 混乱**：Claude Code 自动启动服务时无法纳入统一管理

### 目标

一个本地开发专用的进程管理工具，让所有本地服务：

- 统一后台运行，不占用终端
- 集中查看状态、端口、日志、错误
- 端口由工具统一分配，消除冲突
- Mac 重启后可自动恢复关键服务

```

│                        用户侧                            │
│                                                         │
│   CLI (nanny)              Web Console (浏览器)          │

│   nanny logs foo                                        │
└────────────┬───────────────────────┬────────────────────┘
             │  Unix socket IPC      │  HTTP / SSE
             ▼                       ▼
┌─────────────────────────────────────────────────────────┐
│                      Daemon 进程                         │
│                                                         │
│  ┌─────────────┐  ┌──────────────┐  ┌───────────────┐  │

│  └──────┬──────┘  └──────────────┘  └───────────────┘  │
│         │                                               │
│  ┌──────┴──────────────────────────────────────────┐    │
│  │           受管子进程                              │    │
│  │  parquet-explorer/backend  :5001                 │    │
│  │  parquet-explorer/frontend :5173                 │    │
│  │  md-viewer/server          :3000                 │    │
│  │  md-viewer/rag             :3001                 │    │
│  └──────────────────────────────────────────────────┘    │
└─────────────────────────────────────────────────────────┘
             │
             ▼
    launchd (macOS LaunchAgent)
    开机自动拉起 daemon
```

### 核心组件

| 组件 | 职责 |
|---|---|
| **Daemon** | 常驻后台，管理所有子进程，持有端口注册表，捕获日志，提供 HTTP API |
| **CLI** | 用户控制入口，通过 Unix socket 与 daemon 通信 |
| **Web Console** | daemon 内嵌 serve 的静态页面，浏览器查看状态和实时日志 |

## 3. 项目配置：app-nanny.toml

每个受管项目在根目录放一个 `app-nanny.toml`，描述服务元信息。支持两种模式：

### 模式 A：简单模式（整体托管）

适合已有完整 justfile 的存量项目，零改造成本：

```toml
name = "parquet-explorer"
command = "just dev"          # 默认就是 "just dev"，可省略
autostart = false             # daemon 启动时是否自动拉起（默认 false）
restart = "on-failure"        # "always" | "on-failure" | "never"
max_restarts = 5              # 防止无限重启

[ports]
PORT = 5001                   # 环境变量名 = 端口号，daemon 启动时注入
VITE_PORT = 5173
```

app-nanny 将 `[ports]` 表里的键值对直接作为环境变量注入：
```bash
PORT=5001 VITE_PORT=5173 just dev
```

这样 justfile 里的变量名是什么，toml 里就写什么，不做任何魔法转换。

对应 justfile 改造方式（建议但不强制）：
```just
dev:
    PORT=${PORT:-5001} python backend/app.py &
    cd frontend && npm run dev -- --port ${VITE_PORT:-5173}
```

### 模式 B：精细模式（多进程独立管理）

适合需要单独重启、独立查看日志的多服务项目：

```toml
name = "md-viewer"
autostart = true
restart = "on-failure"

[processes.server]
command = "bun --watch run src/server.ts"
port = 3000                   # 精细模式每个 process 各自收到 PORT=<port>

[processes.rag]
command = "bun --watch run src/rag-server.ts"
port = 3001
memory_warn_mb = 512          # 内存超出时告警

[processes.translate]
command = "python3 translate-server/server.py"
port = 3002
```

两种模式可以随时从 A 升级到 B，不影响全局注册表。

### 全局注册项目

```bash
cd /workspace/walle/ds-query/parquet-explorer
nanny add .          # 读取 app-nanny.toml，写入全局索引

nanny remove parquet-explorer   # 取消注册（不删除 toml 文件）
```

全局索引存储位置：`~/.local/share/app-nanny/registry.json`（只存项目路径，不存端口）

## 4. 端口管理

### 分配策略

1. **启动前**：daemon 从各项目的 `app-nanny.toml` 读取声明端口，检查是否与其他受管服务冲突
2. **冲突时**：拒绝启动并报错，提示冲突来源（不做自动漂移，避免意外）
3. **注入方式**：Mode A 将 `[ports]` 表的键值对原样注入为环境变量；Mode B 每个 process 各自收到 `PORT=<其声明端口>`
4. **实际发现**：进程启动后，用 `lsof -p <pid> -iTCP -sTCP:LISTEN` 扫描真实绑定端口
5. **不一致告警**：声明端口 ≠ 实际端口时，在 `ps` 输出中标黄警告

### 端口注册表

**只存在于内存中，不持久化**。daemon 每次启动时从 registry.json 拿到项目路径，再读各项目的 `app-nanny.toml` 重建端口表。

这样端口的唯一真相来源始终是各项目的 toml 文件，不存在"ports.json 与 toml 不一致"的问题。

## 5. 日志管理

### 存储位置

```
~/.local/share/app-nanny/logs/
  parquet-explorer.log          # 模式 A：整体日志
  md-viewer-server.log          # 模式 B：每个 process 独立文件
  md-viewer-rag.log
  md-viewer-translate.log
```

### 轮转策略

- 单文件上限：50MB
- 保留文件数：3（即最多 150MB/service）
- 轮转方式：rename + 新建（不用 logrotate 外部依赖）

### 捕获机制

Daemon 启动每个子进程时接管其 stdout/stderr，底层是 OS pipe。每个进程对应一个 goroutine 专门做读取：

```
子进程 stdout/stderr → pipe → goroutine(pipe.Read) → 写日志文件 + 错误检测
```

`pipe.Read()` 是**阻塞调用**：pipe 里没有数据时，goroutine 挂起，CPU 占用为零；子进程写入一行日志，OS 立即唤醒 goroutine 来处理。这是纯事件驱动，不需要轮询，进程沉默时开销完全为零。

### 实时查看

```bash
nanny logs parquet-explorer          # 最近 100 行
nanny logs parquet-explorer -f       # 实时追
nanny logs md-viewer/rag -f          # 精细模式指定 process
```

## 6. 错误捕获

daemon 实时扫描所有进程的 stdout/stderr，识别以下错误模式：

| 模式 | 识别特征 |
|---|---|
| HTTP 500 | 日志行含 ` 5[0-9][0-9] `（HTTP 状态码） |
| Python traceback | `Traceback (most recent call last):` 开始的多行块 |
| Node/Bun stack trace | `Error:` 后跟 `    at ` 开头的多行 |

**事件分组**：将触发行及其后的连续相关行（traceback/stack）打包为一个"错误事件"，存入内存 ring buffer（最近 50 条），同时写入独立错误日志 `~/.local/share/app-nanny/logs/<service>.error.log`。

### 错误查询

```bash
nanny errors parquet-explorer           # 列出最近错误事件
nanny errors parquet-explorer --last    # 最后一条完整事件
nanny errors parquet-explorer --copy    # 复制最后一条到剪贴板（pbcopy）
nanny errors parquet-explorer -f        # 实时监听新错误
```

## 7. 进程监控与统计

### 采集策略：自适应采样

daemon 是常驻进程，设计上要低功耗。采样频率根据是否有"活跃观察者"自动切换：

| 模式 | 触发条件 | 采样间隔 |
|---|---|---|
| **空闲模式** | 无 CLI 连接、web console 未打开 | 内存 RSS 每 30s，CPU 不采集 |
| **活跃模式** | 有 `nanny ps` / `nanny logs` 在运行，或 web console 已打开 | 内存 RSS 每 5s，CPU% 每 5s |

**RSS 采样的实际成本**：用 Go 原生 syscall（`proc_pidinfo`）读取，单次约 50 微秒。10 个进程、5s 间隔时，每小时 CPU 消耗约 360ms，占单核时间的 0.01%，可忽略不计。30s 间隔时约 0.002%。两者差异极小，采用自适应设计主要是架构上更合理（没人看时没必要轮询），而非性能压力。

**CPU% 只在活跃模式下计算**（两次采样差值），空闲时不做 CPU 轮询。

日志捕获（stdout/stderr pipe）和进程状态感知（wait/SIGCHLD）是纯事件驱动，没有轮询，始终低开销。详见第 5 节"捕获机制"。

### 采集内容

**进程级**：
- PID、状态（running / stopped / crashed）
- 启动时间、运行时长（uptime）
- 重启次数、最后重启时间、最后重启原因
- 内存 RSS（`ps -o rss= -p <pid>`，按模式采样）
- CPU%（仅活跃模式）
- 声明端口 vs 实际端口

**服务级汇总**：
- 日志文件大小
- 错误计数（最近 1h / 24h）
- 最后一条错误时间

### 内存告警

当进程 RSS 超过 `memory_warn_mb` 阈值时：
- `nanny ps` 中该行标黄
- daemon 日志记录一条 WARN
- 不自动 kill，只告警

## 8. CLI 命令一览

```bash
# 服务管理
nanny add .                      # 注册当前目录项目
nanny remove <name>              # 取消注册
nanny start <name>               # 启动服务（所有 processes）
nanny start <name>/<process>     # 启动单个 process（精细模式）
nanny stop <name>                # 停止服务
nanny restart <name>             # 重启服务
nanny restart <name>/<process>   # 重启单个 process

# 状态查看
nanny ps                         # 所有服务状态总览
nanny status <name>              # 某服务详细状态

# 日志
nanny logs <name> [-f] [-n 100]
nanny logs <name>/<process> [-f]

# 错误
nanny errors <name> [--last] [--copy] [-f]

# Daemon 管理
nanny daemon start               # 手动启动 daemon
nanny daemon stop                # 停止 daemon
nanny daemon status              # daemon 自身状态
nanny install                    # 写入 launchd，开机自启
nanny uninstall                  # 移除 launchd

# Web 控制台
nanny dashboard                  # 浏览器打开 http://localhost:7070
```

## 9. Web 控制台

### 访问

```bash
nanny dashboard    # 自动打开浏览器
# 或直接访问 http://localhost:7070
```

**端口 7070 默认始终开放**：daemon 启动即监听，一个空闲 TCP socket 占用可忽略不计（~1KB），不影响业务端口管理。

### 功能

**服务总览页**：

```
┌─────────────────────────────────────────────────────────┐
│ nanny                                     ● daemon up   │
├─────────────────────────────────────────────────────────┤
│ PROJECT            PROCESS    PORT   MEM    STATUS      │
│ ▼ md-viewer                                             │
│     server         3000       45M    ✓ 2h14m  [↺] [■]  │
│     rag            3001       128M   ✓ 2h14m  [↺] [■]  │
│     translate      3002       —      ✗ crashed [↺] [▶]  │
│ ▶ parquet-explorer                                      │
│     [just dev]     5001,5173  62M    ✓ 45min  [↺] [■]  │
├─────────────────────────────────────────────────────────┤
│ ERRORS  [md-viewer/rag ▼]  [errors only ▼]  🔴 2 errors│
│ [12:01:45] POST /query 500 ◀── 点击展开完整 traceback    │
│ [11:58:12] POST /query 500 ◀── 点击展开完整 traceback    │
└─────────────────────────────────────────────────────────┘
```

**日志区默认展示"错误事件"（errors only），不展示全量请求日志**，原因：

- 全量日志（每条 200/304 都显示）会迅速淹没页面，查错时信噪比极低
- 全量日志已经在文件里，`nanny logs <name> -f` 在终端看更合适

切换下拉框可选 `all logs`，届时展示最近 200 行（SSE 流式推送），超出自动丢弃旧行，页面不会无限增长。

### 技术实现

- **静态文件**：HTML + 原生 JS（无框架），用 Go `embed` 打包进二进制
- **实时日志**：SSE（Server-Sent Events），单向推送，无需 WebSocket
- **数据 API**：daemon 的 HTTP API，CLI 和 Web Console 共用同一套
- **无需单独部署**：web console 随 daemon 一起启动，无构建步骤

## 10. macOS 集成

### launchd 自动启动

```bash
app-nanny install
```

写入 `~/Library/LaunchAgents/com.app-nanny.daemon.plist`：
- 用户登录后自动启动 daemon
- Daemon crash 后自动重启
- 日志写入 `~/Library/Logs/app-nanny-daemon.log`

### 开机后服务恢复

daemon 重启后：
1. 读取 `registry.json` 恢复项目列表
2. 只自动启动 `autostart = true` 的服务
3. 其余服务保持 stopped，等待用户手动启动

## 11. 实现语言与技术栈

| 层 | 技术 | 理由 |
|---|---|---|
| Daemon + CLI | **Go** | 原生单二进制、低内存、goroutine 适合多进程管理、os/exec 成熟 |
| IPC | Unix socket + JSON-RPC，路径 `~/.local/share/app-nanny/app-nanny.sock` | 简单可靠，CLI 和 daemon 同机通信 |
| Web console | 原生 HTML/JS，Go `embed` | 零依赖，打包进二进制，无构建步骤 |
| 日志存储 | 本地文件 + ring buffer（内存） | 无需数据库 |
| 状态持久化 | JSON 文件 | 轻量，人可读，便于调试 |
| 发布 | `go build` → 单二进制 | 复制到 PATH 即可用 |

## 12. 不做的事（明确 Out of Scope）

- **请求代理 / APM**：不做反向代理，不统计请求耗时、吞吐量
- **远程管理**：仅限本机，不支持 SSH 远程或多机管理
- **容器管理**：v1 不管 Docker 容器，只管本地进程。
  未来扩展成本中等：Docker 有标准 API，`docker ps/logs/start/stop` 可以直接 shell out，但端口映射（容器内端口 ≠ 宿主机端口）和生命周期（容器 stopped 但仍然存在）与本地进程模型不同，需要新的抽象层，估计 +30% 工作量。作为 v2 feature 预留。
- **语言工具链感知**：不解析 package.json / pyproject.toml 等，只执行命令
- **自动端口漂移**：端口冲突时报错，不自动换端口（避免意外）
- **内存 heap profiling**：`memory_warn_mb` 监控的是 **RSS（Resident Set Size）**，即进程当前实际占用的物理内存，一个数字，由 OS 直接提供。Heap profiling 是深入分析进程内存内部结构（哪些对象占了多少、是否有内存泄漏），需要语言运行时配合（Python `tracemalloc`、Node.js `--inspect`）——前者是"体重秤"，后者是"体检报告"。nanny 只做体重秤。
